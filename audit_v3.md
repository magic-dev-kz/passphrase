# PassPhrase v2 -- QA Audit Report

**Auditor:** Nash, QA -- OpenClaw
**Date:** 2026-03-29
**Files:** `index.html`, `app.js`, `style.css`, `sw.js`

---

## Overall Score: 8 / 10

**Verdict: PASS with reservations.** Solid security foundation, good accessibility, clean code. A few issues need fixing before production -- one P1, several P2s.

---

## 1. Security: Passwords never in localStorage (PASS)

- `history` array is declared as a plain in-memory variable (`let history = [];` -- app.js:59).
- No `localStorage.setItem` call ever touches passwords. Confirmed via full text search.
- `localStorage` is used ONLY for theme preference (`pp-theme`) -- acceptable.
- Comment on line 381 and line 770 explicitly documents the intent: "session-only -- never stored in localStorage".
- History panel footer warns user: "Session only -- clears when you close the tab".

**Result: PASS**

## 2. crypto.getRandomValues -- no Math.random (PASS)

- `secureRandomInt()` (app.js:10-19) uses `crypto.getRandomValues(new Uint32Array(1))`.
- Implements rejection sampling to eliminate modulo bias (`maxValid` guard) -- correct.
- All generation paths (`generatePhrase`, `generateClassic`, `generatePin`) use `secureRandomElement` / `secureRandomChar` which delegate to `secureRandomInt`.
- Zero occurrences of `Math.random` in the entire codebase.

**Result: PASS**

## 3. Strength Checker: Entropy & Dictionary Check (PASS with notes)

### Entropy calculation
- `analyzePassword()` (app.js:659-691): uses `len * log2(charsetSize)` for character-based entropy. Standard approach.
- For passphrase mode, `calculateEntropy()` (app.js:220-237) correctly uses `wordCount * log2(dictSize)` plus additive bits for number/symbol options. This is the right model for Diceware-style passphrases.

### Dictionary check
- Checks against a hardcoded list of 10 common passwords (app.js:672).
- Uses `string.includes()` so partial matches work (e.g., "mypassword123" is caught).
- Sequential pattern detection via regex (`/^(.)\1+$/` for repeated chars, sequential digits/letters).
- Penalization: dictionary match reduces entropy to 30%, sequential to 40%.

### Issues found
- **P2-ENTROPY-01:** The checker's `analyzePassword()` always uses character-level entropy, even if the user pastes a passphrase generated by the app. It does NOT get the benefit of the word-level entropy calculation from `calculateEntropy()`. These are two separate functions with different logic. The checker could undervalue or overvalue passphrases depending on length.
- **P3-DICT-01:** Dictionary list is only 10 words. Consider using a larger common-password set (e.g., top 100-200). Current set misses "iloveyou", "sunshine", "football", "shadow", etc.

**Result: PASS (functional, minor improvements possible)**

## 4. Auto-clear Clipboard (PASS with one issue)

- `scheduleClipboardClear()` (app.js:341-347): sets a 30-second `setTimeout` to write empty string to clipboard.
- Toast with countdown (app.js:349-379): shows visual countdown from 30 to 0, bar animates, toast auto-hides.
- Timer is properly reset if user copies again before 30s expires (`clearTimeout` on line 342).

### Issues found
- **P2-CLEAR-01:** Clipboard is NOT cleared on tab close / `visibilitychange` / `beforeunload`. The 30s timer only runs while the page is open. If a user copies a password, then closes the tab within 30 seconds, the password remains in the system clipboard indefinitely. Should add a `beforeunload` or `visibilitychange` listener to clear clipboard immediately.

**Result: CONDITIONAL PASS**

## 5. WCAG AA Contrast (PASS with one issue)

### Dark theme (default)
- Primary text `#FFFFFF` on `#1C1C1E` -- ratio ~16.75:1 (PASS AAA)
- Secondary text `#A1A1A6` on `#1C1C1E` -- ratio ~5.5:1 (PASS AA)
- Muted text `#8E8E93` on `#1C1C1E` -- ratio ~4.5:1 (PASS AA, borderline)
- Accent `#0A84FF` on `#1C1C1E` -- ratio ~4.5:1 (PASS AA for large text, borderline for normal)

### Light theme
- Primary text `#1C1C1E` on `#F2F2F7` -- ratio ~15.4:1 (PASS AAA)
- Secondary text `#636366` on `#FFFFFF` -- ratio ~5.9:1 (PASS AA)
- Muted text `#6C6C70` on `#FFFFFF` -- ratio ~5.3:1 (PASS AA)

### Issue
- **P3-CONTRAST-01:** Shield label level 3 uses `#FFD60A` (yellow) for text. On dark background `#2C2C2E` (bg-elevated pill), ratio is approximately 8.6:1 -- fine. But on light theme `#FFFFFF` background, yellow `#FFD60A` on white is approximately 1.4:1 -- FAILS AA badly. The "Strong" level is unreadable in light mode.

**Result: CONDITIONAL PASS (light-mode yellow is a problem)**

## 8. XSS: escapeHtml / textContent (PASS with one issue)

- Password display uses `$passwordText.textContent = text` (app.js:198, 210) -- safe.
- Shield label/time use `.textContent` (app.js:307-309) -- safe.
- Checker verdict/details use `.textContent` (app.js:697, 704) -- safe.
- History: `toggleHistoryPassword` uses `.textContent` (app.js:433, 437) -- safe.
- `escapeAttr()` function (app.js:443-445) escapes `& " ' < >` for HTML attribute contexts.

### Issue
- **P1-XSS-01:** In `renderHistory()` (app.js:402-411), the history item is built via string concatenation with `innerHTML`. The password is placed into a `data-password` attribute using `escapeAttr()`. However, `escapeAttr` does NOT escape backticks (`` ` ``). While backtick-based attribute injection is a legacy IE issue and not exploitable in modern browsers, the more serious concern is that `formatDate()` output is injected via innerHTML without escaping. Although `formatDate` only produces digits and dots from a controlled ISO string, the overall pattern of building DOM via innerHTML string concatenation is fragile.

  **More critically:** the `data-password` attribute on the copy button (line 408) means the password is present in the DOM as a readable HTML attribute. Any browser extension, CSS exfiltration attack, or `document.querySelectorAll('[data-password]')` call can harvest all passwords from the DOM. The password should be retrieved from the closure/array index, not stored in a DOM attribute.

**Result: CONDITIONAL PASS (P1 -- password in DOM attribute)**

## 9. Service Worker Version (ISSUE)

- Cache name: `passphrase-v2` (sw.js:1).
- This is v2 of the app with new features (history, checker, toast), but the SW cache name is still `passphrase-v2`.
- **P2-SW-01:** If this is a version bump release (v2 features are new), the cache name should reflect the build/deploy version (e.g., `passphrase-v2.1` or use a hash). If returning users had the old v2 cache, the SW will NOT invalidate it because the cache name is identical. New code will not be served until the user manually clears cache.

**Result: FAIL (cache name not bumped)**

## Summary of Issues

| ID | Severity | Description |
|----|----------|-------------|
| P1-XSS-01 | **P1** | Password stored in DOM `data-password` attribute -- accessible to extensions/CSS exfil. Remove attribute, use array index reference instead. |
| P2-CLEAR-01 | **P2** | Clipboard not cleared on tab close. Add `beforeunload` listener to call `navigator.clipboard.writeText('')`. |
| P2-ENTROPY-01 | **P2** | Checker and generator use different entropy functions. Checker undervalues passphrases. Consider unifying or adding passphrase detection to `analyzePassword()`. |
| P2-SW-01 | **P2** | SW cache name `passphrase-v2` not bumped for new release. Returning users will get stale assets. |
| P3-DICT-01 | **P3** | Dictionary check only has 10 entries. Expand to top 100+ common passwords. |
| P3-CONTRAST-01 | **P3** | Yellow strength label (`#FFD60A`) fails WCAG AA on light theme white background (~1.4:1 ratio). Use a darker gold or add a dark text shadow. |

## What Works Well

- Clean IIFE encapsulation -- no globals leak.
- Rejection sampling in `secureRandomInt` eliminates modulo bias -- textbook correct.
- Focus management (trap, restore, roving tabindex) is thorough.
- `prefers-reduced-motion` handled in both CSS and JS.
- History is genuinely session-only -- verified no persistence path exists.
- Toast countdown with visual progress bar is polished UX.
- `aria-live="polite"` on password display and crack time.
- Fallback copy mechanism for older browsers.

**Final Verdict:** 8/10 -- Fix P1-XSS-01 (password in DOM) and P2-CLEAR-01 (clipboard on tab close) before shipping. The rest can go into a follow-up patch.
